# 长期 Vibe coding 后，我是怎么管理 Agent 的上下文的？


![ChatGPT Image 2026年7月2日 12_30_01.png](https://files.seeusercontent.com/2026/07/02/aO4j/ChatGPT-Image-202672-12_30_01.png)
如果你也做过一段时间 Vibe coding，应该会有一种很熟悉的感受：

有时候模型真的很聪明，甚至聪明到让你惊喜；

但有时候，它又会突然“忘事”、跑偏、误解你的意图，最后写出一份看起来很努力、实际上完全不对路的结果。

后来我慢慢发现，问题往往不在模型本身，而在于——**我们到底有没有把上下文管理好**。

这件事一旦想明白，Agent 的表现会稳定很多。

---

## 1. 什么叫 Agent 的上下文管理？

简单说，就是：

> **把有限的上下文预算，分配给最值得模型理解的信息。**

Agent = 大模型（LLM） + 基座工程（Harness Engineering）。

大模型能接收的信息是有限的，而且它并不是“信息越多越聪明”。

实际使用下来，模型在**中等信息量**下往往表现最好：

- 信息太少，它会缺少约束，容易乱猜；
- 信息太多，它会抓不住重点，容易遗忘；
- 信息刚刚好，它才最容易稳定输出。

所以，真正好的上下文管理，不是把所有内容都堆给模型，而是要让它**在合适的时间，看到合适的内容**。

---

## 2. 我们把 Agent 想成一个新来的实习生

为了更直观地理解，我一般会把 Agent 想成一个“新来的实习生”。

每开一个新的 Agent 对话，就像新招了一位实习生：

- 他不认识项目；
- 他不知道仓库结构；
- 他也不清楚哪些地方能动，哪些地方不能动。

而他工作的地方，是一个固定的工作台。

这个工作台上有一份说明书，叫 **`AGENTS.md`**。

它的作用不是把所有知识都塞进去，而是告诉模型：

- 你是谁；
- 你在哪工作；
- 先看什么；
- 什么不能乱动；
- 遇到问题该怎么找资料。

换句话说，`AGENTS.md` 是工作台说明书，不是百科全书。

---

## 3. 要让新来的实习生快速上手，靠的不是临时培训

如果每来一位实习生都要重新口头培训一遍，那效率会非常低。

更好的办法，是把仓库里的信息提前整理成几类文档：

### 3.1 仓库的整体结构

先让他知道：

- 这个仓库分哪些区域；
- 哪些目录放什么；
- 资料、工具、说明书分别在哪里。

这样他需要什么，就能自己去找。

你可以把这部分理解为 `STRUCTURE.md` 一类的文档。

### 3.2 项目的设计原则和实现逻辑

然后再告诉他：

- 这个项目是做什么的；
- 核心思路是什么；
- 技术栈是什么；
- 扩展时要遵守什么规则。

这一类内容，适合放在 `ROADMAP.md`、`CHANGELOG.md` 这样的文档里。

### 3.3 项目当前的状态

最后还要告诉他：

- 哪些功能已经实现；
- 哪些模块还在路上；
- 哪些说明已经写好；
- 当前项目到底推进到哪一步了。

这类信息，适合放在 `PROJECTSTATE.md` 里。

这样一来，新来的实习生不需要“猜”，也不会因为信息断层而乱做决定。

---

## 4. 真正干活时，靠的是任务卡

光知道仓库结构还不够。

真正开始做事的时候，还需要一张**任务卡**。

我现在越来越相信一件事：

> **Agent 不是靠“临场发挥”完成任务的，而是靠“任务边界”完成任务的。**

一张好的任务卡，至少应该包含下面这些内容：

```shell
# 项目背景
这次任务处于什么阶段，前后文是什么。

# 任务目标
这次到底要完成什么，验收标准是什么。

# 能做的事
允许改哪些东西，避免误伤已经做好的部分。

# 不能做的事
哪些事情绝对不能碰，提前画好红线。

# 能修改、增添 & 删除的文件
明确文件边界，减少无谓改动。

# 实现要求
具体实现时要遵守哪些规则。

# 测试要求
任务完成后如何验证结果。

# 需要更新的文档
把改动记录下来，方便后来的“新实习生”快速接手。

# 验收标准
什么样的结果才算真正完成。

# 完成后需要运行的命令
收尾动作要写清楚，别留下施工现场。
```

这套结构的好处在于：

- 任务目标不会丢；
- 约束条件不会丢；
- 验收标准不会丢；
- 后续复盘也不会丢。

对长期项目来说，这比“让模型自己悟”可靠得多。

---

## 5. 为什么这套方法会越来越重要？

因为我们现在面对的不是“一次性问答”，而是**长期协作**。

长期协作最怕什么？

不是模型不够聪明，而是：

- 上下文混乱；
- 任务边界不清；
- 文档分散；
- 信息没有沉淀。

一旦这些问题同时出现，Agent 就很容易变成“看起来很能干，实际上很难控”。

而当你把这些信息拆清楚之后，模型的表现反而会更稳：

- 它知道先看什么；
- 它知道哪些内容是背景；
- 它知道哪些内容是硬约束；
- 它知道这次任务到底该怎么收口。

这就是上下文管理的价值。

---

## 6. 我的一个小结

如果让我把 Agent 上下文管理浓缩成一句话，那就是：

> **不要把模型当成会读心术的神明，而要把它当成一位需要被正确引导的新同事。**

你给它的不是“所有信息”，而是“正确的信息”；

你要做的不是“无限喂料”，而是“有结构地组织信息”。

这也是我在长期使用 Agent 之后，最深的一点体会：

模型的上限固然重要，但更重要的，往往是我们怎么安排上下文、怎么设计任务、怎么沉淀文档。

当这套系统搭起来之后，Agent 才真正像一个能长期协作的工作伙伴。
