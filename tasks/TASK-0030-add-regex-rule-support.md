# TASK-0030: Add Regex Rule Support

## Status

Planned

## Goal

Add deterministic regex-based review rules configured from review profiles.

The project already supports several built-in deterministic rules and has a documented rule metadata / execution registry boundary. TASK-0030 should introduce configurable `regex_rules` so users can define their own pattern-based checks without changing Python code.

This task adds a new deterministic rule capability. It must not introduce LLM review.

## Background

Previous tasks established the current rule foundation:

* TASK-0026 added `docs/RULES.md`
* TASK-0027 consolidated legacy rule documentation
* TASK-0028 added the rule metadata registry
* TASK-0029 documented the boundary between rule metadata and rule execution registries

Current built-in rule IDs include:

```text id="ih801p"
forbidden_terms
absolute_claims
markdown_structure
markdown_links_images
```

The next useful capability is configurable regex matching.

Examples of user needs:

* flag exaggerated claims
* flag risky marketing phrases
* flag missing disclaimer patterns
* flag platform-specific wording
* flag unwanted formatting conventions
* flag domain-specific terms that are easier to express with regular expressions than exact terms

Example profile direction:

```yaml id="6172wu"
regex_rules:
  - id: exaggerated_claims
    pattern: "唯一|第一|最强|绝对|100%"
    severity: warning
    message: "Avoid absolute or exaggerated claims."
    suggestion: "Use a more cautious and evidence-based expression."
```

### 2. Add Regex Rule Profile Model

Add support for a new optional profile section:

```yaml id="23jxyq"
regex_rules:
  - id: exaggerated_claims
    pattern: "唯一|第一|最强|绝对|100%"
    severity: warning
    message: "Avoid absolute or exaggerated claims."
    suggestion: "Use a more cautious and evidence-based expression."
```

Suggested fields:

```text id="nqmoww"
id: required string
pattern: required string
severity: required severity
message: required string
suggestion: optional string
case_sensitive: optional boolean, default false or true based on project convention
```

Recommended default:

```text id="9k41h7"
case_sensitive: false
```

because many content-review use cases are user-facing text and case-insensitive matching is usually safer for English text. Chinese matching is unaffected in most cases.

However, if the current project convention favors explicit behavior, use:

```text id="036t1f"
case_sensitive: true
```

and document it clearly.

The selected default must be documented and tested.

### 3. Define Finding Rule ID Behavior

Each regex rule should produce findings using the configured regex rule `id` as the finding `rule_id`.

Example:

```yaml id="pl8tpk"
regex_rules:
  - id: exaggerated_claims
    pattern: "唯一|第一|最强"
    severity: warning
    message: "Avoid exaggerated claims."
```

If the Markdown contains:

```markdown id="vp16ch"
这是最强的解决方案。
```

The finding should use:

```text id="b1lbpa"
rule_id = exaggerated_claims
```

not:

```text id="w7t28i"
rule_id = regex_rules
```

Reason:

* suppression should be precise
* rule counts should be meaningful
* quality gate matched findings should identify the configured rule
* reports should show the user-defined rule ID

### 4. Validate Regex Rule IDs

Profile validation should ensure that each regex rule `id` is:

* non-empty
* stable as a string
* unique within `regex_rules`
* safe to use as a `rule_id`

Recommended allowed pattern:

```text id="i7flvu"
^[a-z][a-z0-9_]*$
```

Examples of valid IDs:

```text id="uryk79"
exaggerated_claims
medical_guarantee_claims
wechat_sensitive_phrase
```

Examples of invalid IDs:

```text id="h78t05"
ExaggeratedClaims
exaggerated-claims
regex rule
123_rule
```

If the project already has a rule ID convention, follow the existing convention instead.

### 5. Validate Regex Patterns

Profile validation should compile regex patterns and report invalid patterns clearly.

Invalid pattern example:

```yaml id="mj1tw8"
regex_rules:
  - id: broken_regex
    pattern: "["
    severity: warning
    message: "Broken regex."
```

Expected behavior:

* profile validation should fail according to existing validation semantics
* review should not run with an invalid regex profile
* error message should identify the invalid regex rule ID and pattern problem
* no traceback should be shown for normal user errors

### 6. Define Matching Behavior

Regex rules should scan Markdown text deterministically.

Recommended initial behavior:

* scan raw Markdown text line by line
* produce one finding per regex match
* include `line`
* include `column`
* include `matched_text`
* include `context` if the current finding model supports it
* do not match across multiple lines in TASK-0030
* do not modify the input document
* do not perform automatic rewrite
* do not attempt semantic interpretation

Line and column behavior should follow existing project conventions.

If existing findings use 1-based line and column positions, regex findings should do the same.

### 7. Respect Existing Suppression Behavior

Regex rule findings must support existing inline suppression syntax.

Examples:

```markdown id="53t9yg"
这是最强的解决方案。 <!-- content-review-disable-line exaggerated_claims -->
```

```markdown id="y38b3x"
<!-- content-review-disable-next-line exaggerated_claims -->
这是最强的解决方案。
```

Do not introduce new suppression syntax.

Do not change suppression behavior for existing rules.

Possible naming:

### 10. Update Rule Registry Metadata Carefully

TASK-0028 introduced a metadata registry for built-in rule metadata.

For TASK-0030, document the distinction carefully:

* `regex_rules` is a profile-configured rule type
* individual regex findings use the configured regex rule `id`
* user-defined regex rule IDs are not necessarily listed in the built-in metadata registry
* if adding a metadata entry for the regex rule type, do not confuse it with user-defined finding rule IDs

Recommended approach:

Add metadata for the built-in capability only if it fits the existing registry model:

```text id="o2lxue"
regex_rules
```

with a description such as:

```text id="060fne"
Profile-driven regex rule capability. Individual findings use the configured regex rule id.
```

If the current metadata registry is intentionally only for built-in finding rule IDs, do not add `regex_rules` to it. Instead, document that profile-defined regex rule IDs are dynamic.

Whichever approach is chosen must be documented and tested.
