# TASK-0024: Improve Markdown Report Output

## Status

Planned

## Goal

Improve Markdown output for single-file and batch review results so the generated `.md` report is easier to read, share, and use as a CI artifact.

After this task, users should be able to run:

```bash
content-review review article.md --profile profiles/my-wechat.yaml --format markdown --output review-report.md
```

or:

```bash
content-review batch articles --profile profiles/my-wechat.yaml --recursive --fail-on error --format markdown --output batch-report.md
```

and get a structured Markdown report with:

* Clear summary
* Severity counts
* Rule counts
* Quality gate status when `--fail-on` is used
* Detailed findings
* Per-file sections for batch review
* Useful empty-state output when no findings exist

This task improves reporting and human review workflows without changing rule behavior or JSON schemas.

## Background

The project currently supports:

* Markdown reading
* YAML `ReviewProfile` loading
* `forbidden_terms` rule
* `absolute_claims` rule
* `allow_terms`
* Inline suppression comments
* Single-file review
* Batch review
* Text / JSON / Markdown output
* CLI quality gate via `--fail-on`
* CI-friendly exit codes
* `content-review profile validate`
* `content-review profile init`
* `content-review profile list`
* Built-in example profiles
* GitHub Actions CI example

The existing Markdown output is functional, but it is not yet optimized as a report that humans can review after a CI run or before publishing.

Now that the project can be used in automated workflows, Markdown reports should become more useful as review artifacts.

## Exit Code Behavior

This task must preserve existing exit code behavior.

```text
0 = command completed and quality gate passed, or no quality gate configured
1 = command completed but findings met or exceeded the --fail-on threshold
2 = command/configuration/input/output error
```

Markdown report rendering must not change how exit codes are calculated.

If `--fail-on error` fails, the Markdown report should still be written when possible before the command exits with code `1`.

## Single-File Markdown Report Design

For `content-review review`, the Markdown output should have a clear structure.

Recommended structure:

```markdown
# Content Review Report

## Summary

| Field | Value |
| --- | --- |
| File | `article.md` |
| Total Findings | 2 |
| Quality Gate | Failed |
| Fail On | `error` |
| Matched Gate Findings | 1 |

## Severity Counts

| Severity | Count |
| --- | ---: |
| critical | 0 |
| error | 1 |
| warning | 1 |
| info | 0 |

## Rule Counts

| Rule | Count |
| --- | ---: |
| forbidden_terms | 1 |
| absolute_claims | 1 |

## Findings

| Severity | Rule | Line | Column | Message | Suggestion |
| --- | --- | ---: | ---: | --- | --- |
| error | forbidden_terms | 3 | 5 | Forbidden term found: "违规词" | Remove or replace the term. |
| warning | absolute_claims | 8 | 1 | Possible absolute or over-promising claim: "全网最强" | Use a more qualified expression. |
```

Exact field names may follow the current data model.

If column information is not available, use `-`.

If suggestion is not available, use `-`.


## Quality Gate Section

When `--fail-on` is configured, the report should include quality gate information.

Recommended fields:

```text
Quality Gate
Fail On
Matched Gate Findings
Exit Code
```

Example:

```markdown
## Quality Gate

| Field | Value |
| --- | --- |
| Status | Failed |
| Fail On | `error` |
| Matched Findings | 2 |
| Exit Code | 1 |
```

If `--fail-on` is not configured, either omit this section or show:

```markdown
## Quality Gate

Not configured.
```

Recommended behavior:

* Include a quality gate row in the summary only when `--fail-on` is configured.
* Optionally include a dedicated `## Quality Gate` section when it improves readability.
* Avoid duplicating too much information.

## Rule Counts

If findings include `rule_id`, Markdown reports should include a rule count table.

Example:

```markdown
## Rule Counts

| Rule | Count |
| --- | ---: |
| forbidden_terms | 3 |
| absolute_claims | 5 |
```

Rules with zero findings do not need to appear.

For no findings:

```markdown
## Rule Counts

No rule findings.
```

## Highest Severity

For batch file summary, include highest severity per file if simple to compute.

Severity ordering should reuse the existing quality gate severity ordering:

```text
info < warning < error < critical
```

Do not duplicate a second severity ranking system.

If highest severity is difficult to compute cleanly in this task, it may be omitted.

Preferred behavior:

```markdown
| File | Findings | Highest Severity |
| --- | ---: | --- |
| `articles/a.md` | 2 | error |
```

## Markdown Escaping

Markdown table cells should be safe enough for common content.

At minimum, escape or replace table-breaking characters:

```text
|
```

Recommended behavior:

* Replace `|` with `\|`
* Replace newlines inside table cells with spaces
* Wrap file paths and simple values in backticks where appropriate

Do not implement a full Markdown sanitizer in this task.

## Deterministic Output

Markdown output should be deterministic.

Rules:

1. Severity rows should appear in canonical order:

```text
critical
error
warning
info
```

or:

```text
info
warning
error
critical
```

Choose one and document it.

Recommended for reports:

```text
critical
error
warning
info
```

because higher risk appears first.

2. Rule counts should be sorted by rule id.
3. Files should preserve the deterministic discovery order already used by batch review.
4. Findings should preserve existing review order.

Do not add timestamps in this task unless the project already has a deterministic way to control them in tests.

## Suppression Behavior

Suppressed findings must remain excluded from Markdown reports.

This task must preserve TASK-0017 behavior:

```text
suppressed findings do not appear in output
suppressed findings do not count toward summaries
suppressed findings do not trigger --fail-on
```

Markdown reports should show only active, unsuppressed findings.

Do not add `--show-suppressed` in this task.
