
# TASK-0018: Add Absolute Claims Rule

## Status

Planned

## Goal

Add a dedicated `absolute_claims` review rule for detecting absolute, exaggerated, or over-promising expressions in publishable Markdown content.

The rule should help identify wording that may be risky in marketing, product, medical, educational, or public-facing articles.

This task should extend the deterministic rule system without introducing LLM-based review.

## Background

The project currently supports:

* Markdown reading
* YAML `ReviewProfile` loading
* `forbidden_terms` rule
* `forbidden_terms.allow_terms`
* Inline suppression comments
* Single-file review
* Batch review
* Text / JSON / Markdown output
* CLI quality gate via `--fail-on`
* CI-friendly exit codes

The existing `forbidden_terms` rule can already detect configured terms, but absolute claims deserve a dedicated rule because they represent a common and reusable content-risk category.

Examples of absolute or over-promising expressions:

```text
全网最强
永久有效
绝对安全
100% 有效
零风险
彻底解决
保证成功
唯一选择
```

A dedicated `absolute_claims` rule gives these findings a stable `rule_id`, clearer messages, better suggestions, and a foundation for future rule-specific improvements.

## Profile Design

The rule should be configured through `ReviewProfile`.

Recommended YAML example:

```yaml
rules:
  - id: absolute_claims
    enabled: true
    severity: warning
    terms:
      - 全网最强
      - 永久有效
      - 绝对安全
      - 100% 有效
      - 零风险
      - 彻底解决
      - 保证成功
    allow_terms:
      - 唯一标识符
```

Expected behavior:

* Terms listed in `terms` should be detected.
* Terms listed in `allow_terms` should not produce findings.
* If `allow_terms` is omitted, existing detection behavior should remain unchanged.
* If `allow_terms` is empty, existing detection behavior should remain unchanged.
* If the rule is disabled, it should not produce findings.

## Minimal Required Configuration

The minimal valid configuration should be:

```yaml
rules:
  - id: absolute_claims
    enabled: true
    terms:
      - 全网最强
```

If `severity` is omitted, use the project’s existing default severity behavior if one already exists.

If there is no existing default severity behavior, use:

```text
warning
```

## Optional Configuration

```yaml
rules:

      - 全网最强
      - 永久有效
    allow_terms:
      - 永久有效
```

Expected behavior:

* `全网最强` should be detected.
* `永久有效` should not be detected.

## Validation Rules

The profile loader should validate `absolute_claims` configuration.

Valid configuration:

```yaml
rules:
  - id: absolute_claims
    enabled: true
    severity: warning
    terms:
      - 全网最强
      - 绝对安全
    allow_terms:
      - 绝对安全
```

Invalid configuration:

```yaml
rules:
  - id: absolute_claims
    enabled: true
    terms: 全网最强
```

Expected behavior:

* `terms` must be a list of strings.
* A string value for `terms` should be rejected.

Invalid configuration:

```yaml
rules:
  - id: absolute_claims
    enabled: true
    terms:
      - 全网最强
    allow_terms: 绝对安全
```

Expected behavior:

* `allow_terms` must be a list of strings.
* A string value for `allow_terms` should be rejected.

Invalid configuration:

```yaml
rules:
  - id: absolute_claims
    enabled: true
    terms:
      - 全网最强
    severity: high
```

Expected behavior:

* Invalid severity should be rejected according to the project’s existing severity validation rules.

## Matching Behavior

For this task, matching should be simple and deterministic.

Rules:

1. Match literal strings only.
2. Do not implement regex.
3. Do not implement wildcard matching.
4. Do not implement fuzzy matching.
5. Do not implement semantic matching.
6. Do not call an LLM.
7. Matching should follow the current `forbidden_terms` matching style as much as possible.
8. Multiple occurrences may produce multiple findings if the current rule system already behaves that way.
9. If the existing `forbidden_terms` rule reports one finding per line or one finding per occurrence, `absolute_claims` should follow the same convention.

## Finding Behavior

A matched absolute claim should produce a finding with:

```text
rule_id = absolute_claims
severity = configured severity
message = clear explanation of the risky absolute claim
location = line-level location when available
suggestion = safer wording guidance
```

Suggested message format:

```text
Possible absolute or over-promising claim: "全网最强"
```

Suggested suggestion format:

```text
Consider using a more qualified expression, such as "较强", "表现较好", "在特定条件下有效", or provide evidence for the claim.
```

The exact wording may be adjusted to match the project’s existing finding style.
